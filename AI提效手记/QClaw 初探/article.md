QClaw 体验报告：是生产力工具，也是待雕琢的“半成品”

网上说的天花乱坠，还是需要实际上手看看使用体验和表现如何，配置的时间和金钱成本又如何。

先说结论，经过我一天多的捣鼓和大概三四天的简单试用，我认为它有用，即确实能够参与到我日常的学习工作，为我提效，至少能 cover 我调教它花的时间；但是还算不上好用，距离好用还有很长的工程距离。

我认为这应该是一个很有潜力的东西，至少我很希望厂商可以在这类工具上面投入更多的精力，也很期待看到它未来的样子。

在这篇文章中，我将会把它看作一个产品，而不是看作一项技术，也就是说我不会去介绍它的 memory 机制等等技术和实现细节，而是从用户的角度去介绍使用体验，介绍我拿它做了什么。

# 缘起

为什么突然想起来折腾个龙虾？
- 我希望能够在手机上处理一些简单的任务，让我能够更好的利用零碎的时间，比如健身、射箭队训练、守望先锋匹配时间等等。实际上 Claude Code 手机版可能更符合我的需求，但是我怕订阅了之后被封号。虽然 Cursor 出了可以在手机上控制电脑上 Agent 的功能，但这发生在我试用 QClaw 之后，只能之后再去体验了。
- OpenClaw 最近很火，我有 AI 焦虑，认为自己有必要去尝试一下相关工具，光是看网上的消息和听周围人的反馈是远远不够的。
- 我们的项目 Crater 有计划接入智能运维、开发 `crater-cli` 等等。我需要研究一下：如果让 OpenClaw 这类工具操控我们的平台，是通过 RESTful API 更合适还是通过 CLI 工具更合适。
- 这几天来杭州出差，受限于网络条件和行程安排，不太方便搞一些复杂和核心的工作，因此折腾折腾这些，写写博客啥的。

为什么选择 QClaw，而不是 OpenClaw 或者其它的龙虾？

- 总会有些“技术洁癖”，喜欢用一些国外的更“纯粹”的一些工具，但现在我发现国内的一些厂商做了一些还蛮有价值的工程工作，帮助技术落地，认为有必要了解下。
- 看起来安全性更强一些，毕竟我要把它安装到我的主力生产机上。
- 觉得 QClaw 对微信的支持会更好一些（实际上不好），这是我平常最常用的通讯工具。
- 之前去腾讯参观过之后还挺喜欢腾讯的。

所以总的来说，我的目标是体验体验，然后努力让它真的能切实的帮我提效，满足我的一些实际的需求，节约我的一些时间，而不是把它玩的花里胡哨，折腾出来一些我根本不需要的东西。

# 使用体验

## 安装

整个安装过程非常简单，直接在官网下载 macOS 版本的 QClaw 桌面版应用程序然后安装就完事了。连接微信也非常方便，很快就可以开始使用，每天腾讯还送 40M Token，很棒。

安装过程非常流畅，不需要做任何复杂乱七八糟的配置，甚至连时间成本都没有付出，我就可以在懒得带笔记本回去的时候通过微信让 QClaw 帮我完成一些小工作了。

**NOTE**：Mac 用户需要给 QClaw 对应目录的权限，比如可以让它操作操作 home 目录和 OneDrive 的目录，会弹窗获取权限，如果此时不在电脑前它会一直卡在这。

## 添加 API Key

每天赠送的量对于轻度使用完全够了，但是我还是买了 Minimax 的 Plus-极速版 Token Plan，比国外顶尖模型笨点是笨点，但是快是真的挺快，不行还能给 Claude Code 用。其实本来想买智谱的 Plan 的，但是实在是抢不到。

QClaw 可以通过 GUI 界面很方便的配置 API Key，选择厂商和模型然后粘贴 API Key 就行了，注意普通的 API Key 与 Coding Plan 的 Key 不同，是不同的选项。GUI 里能各保存一个，多的话可以直接写在 `~/.qclaw/openclaw.json` 里，然后通过 `/model ` 命令切换。

## 通讯工具集成

我主要尝试了微信 ClawBot、微信客服号、QQ 机器人和飞书机器人几种方式，综合体验下来，飞书机器人的使用体验明显优于其他几个，以下是一些我比较需要的功能他们的支持情况。


| 能力/功能         | 微信 ClawBot | 微信客服号      | QQ 机器人        | 飞书机器人               |
|------------------|--------------|--------------|-----------------|-------------------------|
| 复制部分消息       | -            | -            | 支持            | 支持                      |
| 向用户发送文件     | 支持         | -             | 支持            | 支持                       |
| 接收用户发送的文件  | -            | -            | 支持            | 可以读图或者保存文件到本地  |
| 聊天可在电脑端继续  | -            | 支持          | 支持             | 支持                       |
| 查找聊天记录       | -            | -            | 支持            | 支持                       |
| 定时任务/主动发消息 | -           | -             | -               | 支持                       |
| 格式化消息        | -            | -             | 类似 md        | 类似 md，格式更清楚但是更丑 |
| 接受命令          | 支持         | -             | 支持            | 支持                       |
| 收到反馈          | 支持         | 支持           | 支持            | 支持                       |
| 中断生成          | -            | -             | -             | -                          |

其中：
- 复制部分消息：指选中回复消息中的某部分进行复制，而不是复制全部。
- 向用户发送文件：直接给用户发送文件，比如电脑桌面上的某个文件。但这个功能似乎不太稳定。
- 接收用户发送的文件：允许用户发送图片/文件等，读取或者保存到电脑。
- 聊天可在电脑端继续：微信电脑版不显示 ClawBot 的聊天窗口。
- 查找聊天记录：对聊天历史进行搜索。
- 定时任务/主动发消息：设置定时任务，让 QClaw 执行完发送消息到对应的窗口，微信相关的消息发送工具似乎有问题，一直不能成功，QQ 则非常不稳定，我都没有成功设置定时任务。
- 格式化消息：是否能渲染格式化的文本，微信只有纯文本，但是 QQ 和飞书能渲染类似 md 的格式，看上去清楚不少。
- 接受命令：支持例如 `/status` 之类的一些斜杠命令。
- 收到反馈：收到消息，开始生成时的一些反馈，即能够看出来 QClaw 收到消息了。微信 ClawBot 和 QQ 会显示类似正在输入，客服号会发条消息表示收到，飞书则会用一个敲键盘的小表情回复消息，可爱捏。
- 中断生成：目前这几种通讯工具都没办法手动中止生成，如果不小心打错了，消息发出去了，QClaw 一旦收到了，也无法中断 QClaw 的生成。

总的来说，飞书机器人算是一个可以用的产品了，剩下三个基本上都是个很原始很初期的小玩具。QQ 机器人看起来挺全面，但是实际用起来体验最差，经常不响应聊天，但是斜杠命令还有反应。微信 ClawBot 也不太行，超时逻辑似乎有问题，似乎是生成失败或者卡住了但是并发锁（用于确保消息顺序处理）未正常释放，导致整个聊天直接卡住，连斜杠命令都不响应，需要在电脑上重启 QClaw。飞书用起来体验最好，从来没卡住过，最好的一点是收到消息会给消息加上敲键盘小表情，然后生成结束之后会把这个小表情拿掉，这样即使生成失败了超时了，我也能知道当前的状态，发新的消息来研究和解决问题。

但是还要吐槽一下，在 2026 年 04 月 11 日更新 QClaw 后，我无论如何都无法让它通过飞书给我发文件了，很怪，我确定之前可以。

之后就几乎只通过手机飞书使用 QClaw 了。

## 原生接口

QClaw 原生支持的核心能力，很多工具都是通过 `exec` 执行 Shell 命令来调用。

## 定时任务

看起来和 K8s 中的 CronJob 没啥差别。直接让 Agent 帮忙设置和调整就行。

飞书中的定时任务虽然能够正常发送消息，但不知道有啥毛病，手动触发定时任务没有任何问题，但是到点自动触发，调用模型总会产生“当前访问繁忙或今日用量已达上限，请稍后重试或次日再试。”的回复，偶尔能成功，但实际上用量根本没达到上限，不知道是飞书工具的问题还是 Minimax 的问题。

定时任务包含两种模式，分别是 `main` 和 `isolated`。前者我没试过，据它自己说会直接在主对话中触发，包含主对话上下文，就像用户给它发消息一样，会阻塞主对话。后者则是单独起一个会话处理，不包括主对话上下文，处理完的消息会发到主对话中。

## spawn

上次看到这个词还是在操作系统内核里，可以理解为“生成”，包含一些分离出/分裂出/创建出的感觉。

如果需要处理比较复杂的任务，可以让它 spawn 出一个子 Agent 完成工作，主对话继续做别的事情，然后执行完了通过聊天消息通知，有点类似于 Shell 中的 detached。

这也有两种模式，和上面一样的 `isolated` 和 `persistent`（似乎可以把直接对话不 spawn 子 Agent 理解为 `main`），前者每次运行都是独立对话，没有上下文记忆；后者持续用一个 session，保留上下文。

主会话会充当用户和子 Agent 的消息中转。

## 功耗

感觉有必要单独说一下，OpenClaw 和腾讯的包装看起来都不错，没有做很多多余的事情，后台一直运行 QClaw 耗电非常少，不插电一直挂着和它聊也没啥事，不过我并没有测让它后台高强度跑 Agent。

# 我拿它做了什么

我主要在离开我的主力机时使用我的 QClaw。 它能很好的帮我利用起这样一些零碎的时间，比如健身组间歇、射箭等拔箭、守望先锋和战舰世界匹配的时候，我都可以拿起手机和 QClaw 聊两句，让它帮我处理一些简单的任务。这其实很重要，因为这些任务虽然简单，可能用不了多少脑子，但是如果没有 QClaw 的话，他们可能就会占用我为核心复杂任务准备的大块的专注时间。

这些任务主要是一些文档任务和调研任务，基本不涉及代码任务。具体来说，包括按照白天的使用体验，更新我自己用的 Skills；帮我对白天的想法进行调研，找到一些资料；帮我学习一些概念和知识，可能白天发现自己需要一些知识但不懂，但不舍得花大块的工位时间去学它；帮我给已经完成的代码写一些文档；或者帮我进行一些简单的 review 和创建 PR（通过 `gh` 命令行工具）。

之后可能会让它写一些简单的代码，比如帮我做一些自己玩的 APP 或者网站之类的，但目前我不会让它触碰核心业务代码。一方面是需要 SSH 到服务器上，可能比较麻烦我没试过；另一方面我不能接受这些代码脱离我的直接掌控。我必须要能方便的直接看到这些代码，Cursor 基本是我接受的极限了，像 Claude Code 或者 Codex 这种直接看代码不太方便的都接受不了，更何况 QClaw 这种模型能力差这么多的。
